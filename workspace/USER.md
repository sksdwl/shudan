# USER.md 配置指南

## 文件作用

USER.md 是你的"用户画像"，定义你是谁、怎么说话、喜欢什么风格。这决定了我的回复语气、内容深度和沟通方式。

---

## 配置结构

### 1. 基本信息
```markdown
# USER.md - About My Human

- **Name:** 你的名字
- **What to call them:** 怎么称呼你
- **Pronouns:** 他/她/不重
- **Timezone:** 时区
- **Notes:** 备注，写你的核心特点
```

### 2. Context（背景）
```markdown
## Context

Likes:
- 喜欢的回复风格
- 擅长的内容类型

Doesn't like:
- 讨厌的回复风格
- 反感的沟通方式
```

## 配置原则

### 1. 真实性
- 写真实的你，不是你想成为的人
- Notes 里写你的核心特点（技术栈、工作经验、兴趣爱好）

### 2. 具体化
- 不要写"喜欢技术"，写"喜欢 Python 和 Go"
- 不要写"讨厌废话"，写"回复要控制在 200 字以内"

### 3. 一致性
- Likes 和 Doesn't like 不要冲突
- 风格要统一，不要今天严肃明天幽默

### 4. 可衡量
- 喜好要是可执行的，我能做到的
- 比如用代码示例说明，而不是"多加点代码"

## 高级技巧

### 1. 用 Notes 传递隐含信息
```markdown
- **Notes:** 
   - 技术栈：前端 React + TypeScript
   - 工作：5 年后端开发
   - 兴趣：AI 编程助手、开源项目
```

### 2. 用 Context 定义边界
```markdown
## Context

**专业领域：**
- 前端开发（React、Vue）
- 后端开发（Node.js、Python）
- AI 编程（Python、Copilot）

**回复范围：**
- ✅ 技术问题解答
- ✅ 代码审查
- ✅ 架构设计
- ❌ 生活问题（如"今天天气怎么样"）
- ❌ 非技术话题（如"推荐电影"）
```

### 3. 用 Likes 定义回复模板
```markdown
Likes:
- 优先使用代码块展示解决方案
- 关键概念要加粗强调
- 复杂问题分步骤说明
- 提供相关资源和链接

Doesn't like:
- 不确定的问题要主动询问细节
- 代码示例要完整、可运行
```

## 更新建议

当你的需求变化时，更新 USER.md：

1. **换工作了**：更新 Notes 里的职业信息
2. **换技术栈了**：在 Context 里添加新技术
3. **回复风格变了**：修改 Likes 和 Doesn't like
4. **改称呼了**：修改 What to call them

---

## 最佳实践

### 1. 定期回顾
- 每月检查一次配置是否还准确
- 根据实际使用体验调整

### 2. A/B 测试
- 尝试不同的配置，看哪种效果更好
- 比如先配置"简洁"风格，测试一周
- 再改成"详细"风格，对比效果

### 3. 版本控制
- 如果配置文件变化大，可以先备份
- 保存历史版本，方便回滚

## 常见问题

**Q: 配置完多久生效？**
- 立即生效，下次对话就会按新配置回复

**Q: 配置错误怎么办？**
- 不会影响基本功能，但可能导致回复不符合预期
- 检查格式是否符合规范

**Q: 可以有多个配置吗？**
- 只有一个 USER.md，但可以保存不同版本
- 需要时手动切换内容
